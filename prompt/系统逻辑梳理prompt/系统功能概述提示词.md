# 系统功能概述生成提示词

你是一名资深业务分析师、后端架构师和技术文档工程师。请基于当前项目代码，生成一份“系统功能概述”文档。即使没有任何参考模板，也必须仅凭本提示词输出结构完整、功能边界清晰、可供后续研发快速理解系统能力的文档。

## 任务目标

请从项目代码、页面、接口、实体、服务实现和配置中提炼系统功能，形成一份面向研发协作的系统功能概述文档。

文档目标不是接口清单，也不是业务知识大全，而是回答：

- 系统有哪些端、哪些角色、哪些入口。
- 每个角色能使用哪些功能。
- 每个功能模块解决什么业务问题。
- 每个功能模块包含哪些页面、接口和核心数据流。
- 功能之间如何串成完整业务链路。
- 当前哪些功能已经实现，哪些功能只是前端页面/调用存在但后端未实现。
- 后续开发某个功能时需要注意哪些边界和依赖。

## 输入要求

生成前必须主动阅读并分析以下内容：

1. 根目录说明文档，例如 `README.md`、`CLAUDE.md`、业务概述、接口清单、数据模型等。
2. 页面资源：后台页面、前台页面、菜单配置、路由跳转、页面标题和按钮文案。
3. 前端 API 文件：识别页面实际调用的功能接口。
4. Controller：识别已实现后端功能能力。
5. Service/ServiceImpl：识别每个功能背后的业务规则、校验、状态变化、聚合写入、事务边界。
6. Entity/DTO：识别功能涉及的数据对象。
7. Filter/Interceptor/Common：识别登录、权限、统一响应、异常处理、文件、缓存等系统支撑能力。
8. 配置文件：识别端口、静态资源、缓存、文件路径、接口文档入口等。

如果代码注释或参考文档存在乱码，不要照抄乱码；应根据类名、字段名、接口路径、页面文案和业务上下文恢复功能含义。无法确认的内容必须标注“需进一步确认”，不要编造。

## 分析步骤

1. **识别系统定位**
   - 判断系统服务的业务领域。
   - 说明系统面向哪些用户角色。
   - 说明系统包含哪些端，例如后台管理端、移动端、开放 API、定时任务后台等。

2. **识别入口与角色**
   - 从页面目录、登录接口、过滤器和菜单配置中识别角色入口。
   - 输出角色、入口路径、主要功能、登录态或权限要求。

3. **识别功能模块**
   - 按业务功能分组，而不是按包名机械分组。
   - 示例：员工管理、分类管理、商品管理、套餐管理、用户登录、地址簿、购物车、订单、文件、系统支撑。
   - 每个模块说明功能用途、参与角色、页面入口、核心接口、关键数据对象。

4. **绘制功能流转图**
   - 每个核心模块使用 Mermaid `graph LR` 或 `graph TB` 表达用户操作和接口调用。
   - 图中节点应包含页面、接口、关键处理或数据写入。
   - 不要把所有接口堆在一张图里，应按模块拆分。

5. **补充功能说明**
   - 在流程图后用要点说明核心规则。
   - 例如登录态写入、删除前校验、缓存清理、事务写入、数据隔离、状态变化。

6. **识别系统支撑能力**
   - 说明统一返回、登录过滤、当前用户上下文、异常处理、公共字段填充、JSON 序列化、缓存、文件存储、接口文档等。

7. **识别实现状态**
   - 对比前端 API 调用和后端 Controller。
   - 输出功能实现状态总览，区分已实现和未实现/不完整能力。
   - 特别标注“前端已引用但后端未实现”的功能。

8. **整理开发注意事项**
   - 总结后续开发该系统功能时需要遵守的边界。
   - 例如当前用户隔离、缓存失效、事务边界、接口缺口、数据模型影响。

globs:
---

## 模块输出要求

每个功能模块至少包含：

- 模块名称。
- 模块用途。
- 参与角色。
- 页面入口或调用入口。
- 核心接口。
- 核心数据对象。
- 功能流程图。
- 功能说明。
- 当前实现边界。

系统支撑模块至少包含：

- 登录/鉴权。
- 当前用户上下文。
- 统一返回。
- 全局异常。
- 公共字段填充。
- JSON 序列化。
- 缓存。
- 文件上传下载。
- 接口文档入口。

实现状态总览至少覆盖：

- 所有主要业务模块。
- 已实现能力。
- 前端已引用但后端未实现能力。
- 代码中明显不完整的能力。

## 写作风格要求

1. 使用中文，表达直接、准确、工程化。
2. 文档按功能组织，不按代码包名组织。
3. 类名、接口路径、页面路径、字段名、缓存 key 使用反引号。
4. Mermaid 图用于表达功能流，不用于替代文字说明。
5. 不要只罗列接口；要说明功能用途和用户操作链路。
6. 不要照抄乱码注释。
7. 不要编造不存在的功能；无法确认时标注“需进一步确认”。
8. 对前端已有页面或调用但后端缺失的能力，必须明确写入“当前边界”或“实现状态总览”。
9. 如果项目很小，也要保留完整章节，只是内容可以更精炼。

## 质量检查清单

输出前逐项检查：

- 是否说明系统是做什么的。
- 是否列出了系统端和用户角色。
- 是否覆盖了所有主要页面和 Controller 暴露的功能。
- 是否按业务功能分组，而不是按代码目录机械分组。
- 是否为核心模块提供 Mermaid 功能流图。
- 是否说明了每个模块的核心规则和数据对象。
- 是否识别了系统支撑能力。
- 是否列出了前端已引用但后端未实现的功能。
- 是否避免了接口清单式堆砌。
- 是否没有编造代码中不存在的功能。

## 可直接使用的完整指令

请阅读当前项目代码，生成 `{项目名}` 的《系统功能概述》文档。你必须分析后台页面、前台页面、前端 API、Controller、ServiceImpl、Entity/DTO、登录过滤器、通用组件和配置文件，识别系统入口与角色、功能模块、页面与接口链路、核心数据对象、系统支撑能力、已实现能力和未实现边界。输出 Markdown，结构必须包含系统入口与角色、按业务域分组的功能模块、系统支撑能力模块、功能实现状态总览、功能开发注意事项。每个核心模块使用 Mermaid 图表达功能流，并用文字说明关键规则。不要照抄乱码注释，不要编造不存在的功能，无法确认的内容标注“需进一步确认”。
