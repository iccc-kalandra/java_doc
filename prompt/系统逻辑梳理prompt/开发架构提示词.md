# 开发架构文档生成提示词

## 任务目标

请基于当前项目源码，生成一份可以指导后续开发、代码理解和功能落位的《开发架构》文档。文档必须来自项目真实代码和配置，即使没有任何参考模板，也能独立梳理出项目的整体架构、包结构、关键类、技术栈、分层职责、核心机制和新增功能落位规则。

## 输入要求

生成文档前，请优先读取并分析以下内容：

1. 项目构建文件：如 `pom.xml`、`build.gradle`、`package.json` 等，用于识别项目名称、技术栈、框架版本、运行方式。
2. 应用配置文件：如 `application.yml`、`application.properties`、`.env` 等，用于识别端口、数据库、缓存、文件目录、第三方服务。
3. 后端源码目录：重点分析启动类、配置类、过滤器/拦截器、Controller、Service、Mapper/Repository、Entity、DTO、通用工具类。
4. 前端或静态资源目录：识别页面入口、接口脚本、业务页面、组件、样式、图片等资源组织方式。
5. 现有项目文档：如接口清单、数据模型、系统功能概述、开发规范等，用于交叉校验架构描述。

如果某项能力在代码中不存在，不要为了套模板强行补充，应明确写成“当前项目未接入”或“未发现相关实现”。

## 分析步骤

### 1. 识别项目基本信息

从构建文件和配置文件中提取：

- 项目名称、模块名称、包名或 Maven/Gradle 坐标。
- 框架及版本，如 Spring Boot、MyBatis、React、Vue 等。
- 运行端口、应用名、数据库、缓存、文件存储、接口文档地址。
- Java、Node.js、Python 等运行时版本。

### 2. 梳理总体架构

根据项目实际结构总结系统形态，例如：

- 单体应用、前后端分离应用、静态页面 + 后端 API、微服务、多模块工程等。
- 请求从页面或客户端进入后，经过哪些过滤器、控制器、服务层、数据访问层、缓存或外部服务。
- 使用 Mermaid 图表达整体链路，但图中的节点必须来自项目真实代码和配置。

### 3. 梳理包结构与资源结构

输出后端主包结构和前端资源结构：

- 使用目录树展示主要包和资源目录。
- 为每个包或目录说明职责。
- 不要列出无关临时文件、构建产物或 IDE 文件。
- 如果项目有多个模块，需要分别说明每个模块职责。

### 4. 生成关键类路径速查

从源码中识别对开发最重要的类或文件，并用表格说明：

- 启动类或入口文件。
- 配置类。
- 登录、鉴权、拦截、过滤相关类。
- 统一响应、异常处理、上下文、序列化等通用基础设施。
- 主要业务 Controller。
- 主要 Service、Repository/Mapper。
- 核心 Entity、DTO、VO。
- 文件上传、缓存、消息、定时任务等特殊能力相关类。

表格字段建议为：类型、类路径、说明。

### 5. 梳理分层架构

说明项目的分层方式和每层职责：

- 页面层或客户端层。
- 网关、过滤器、拦截器或中间件层。
- Controller/API 层。
- Service/业务层。
- Repository/Mapper/DAO 数据访问层。
- Entity/Model 数据模型层。
- DTO/VO 请求响应模型层。
- Common/Config/Utils 基础能力层。

需要补充典型请求链路，可使用 Mermaid sequenceDiagram 表达。

### 6. 梳理技术栈

基于依赖和代码使用情况输出技术栈表格：

- 技术名称。
- 当前用途。
- 关键配置或典型使用位置。

仅写项目实际依赖或实际使用的技术。依赖存在但未明显使用时，可以注明“依赖存在，当前使用点较少”。

### 7. 梳理核心架构机制

按项目实际情况描述关键机制，例如：

- 登录与鉴权。
- 当前用户上下文。
- 统一响应格式。
- 全局异常处理。
- ORM 与分页。
- 字段自动填充。
- 缓存读写与失效策略。
- 文件上传下载。
- 接口文档。
- 消息监听。
- 定时任务。
- 第三方服务调用。

如果项目没有某些机制，应在“当前不存在的架构能力”中明确列出，避免后续误用。

### 8. 建立业务模块代码归属

将业务模块与代码文件建立对应关系，推荐表格字段：

- 业务模块。
- Controller/API。
- Service。
- Mapper/Repository。
- Entity/DTO。
- 前端页面或接口脚本。

该部分用于后续开发时快速定位改动范围。

### 9. 输出新增功能落位规则

基于现有项目约定给出新增功能时的代码落位规则，例如：

- 新增表时要补哪些 Entity、Mapper、Service、Controller。
- 接口返回、异常处理、分页、事务、缓存应遵循什么约定。
- 新增前端页面、接口脚本、路由或菜单应放在哪里。
- 新增功能后应同步更新哪些文档。

规则必须贴合当前项目，不要写成通用架构口号。

## 输出格式

建议文档结构如下：

```markdown
---
ruleType: Model Request
description: 项目开发架构
globs:
---

# 项目开发架构

## 项目基本信息

## 总体架构图

## 包结构总览

## 前端资源结构

## 关键类路径速查

## 分层架构

## 技术栈

## 核心架构机制

## 业务模块代码归属

## 当前不存在的架构能力

## 新功能落位规则
```

如果项目没有前端资源、没有缓存、没有文件上传等能力，可以删除对应章节或在章节中说明“当前未发现”。

## 写作要求

1. 使用中文输出，表达清晰、具体、可执行。
2. 所有类名、路径、配置项使用反引号标识。
3. 优先使用表格、目录树、Mermaid 图提升可读性。
4. 不要把参考模板中的公司名、模块名、技术组件照搬到当前项目。
5. 不要虚构不存在的服务、消息队列、定时任务、远程调用或中间件。
6. 对不确定内容使用“未发现”“疑似”“需结合运行环境确认”等表述。
7. 文档应服务于后续开发定位，避免只写概念性介绍。
8. 同一类信息只保留一个主位置，避免大段重复。

## 质量检查清单

生成后请检查：

- 项目基本信息是否能从构建文件或配置文件追溯。
- 包结构是否与真实源码目录一致。
- 关键类路径是否真实存在。
- 技术栈是否与依赖和代码使用情况一致。
- 架构图中的节点是否都是项目实际存在的部分。
- 是否明确区分了“已接入能力”和“当前不存在能力”。
- 新功能落位规则是否能直接指导开发。
- 是否遗漏了登录、异常、响应、缓存、文件、分页、接口文档等基础机制。

## 可直接使用的完整提示词

请阅读当前项目源码和配置，生成一份《开发架构》文档。文档需要基于真实代码，而不是照搬模板。请先分析构建文件、应用配置、源码包结构、前端资源目录、启动类、配置类、过滤器/拦截器、Controller、Service、Mapper/Repository、Entity、DTO、通用类和现有项目文档，然后输出项目基本信息、总体架构图、包结构总览、前端资源结构、关键类路径速查、分层架构、技术栈、核心架构机制、业务模块代码归属、当前不存在的架构能力以及新功能落位规则。所有类名、路径和配置项要准确写出；不存在的能力要明确说明未接入，不要虚构；输出使用中文，优先使用表格、目录树和 Mermaid 图，保证后续开发者即使不了解项目，也能根据这份文档快速判断代码位置、调用链路和新增功能应该放在哪里。
