# HTTP 接口清单生成提示词

你是一名资深后端接口分析师和技术文档工程师。请基于当前项目代码，生成一份“HTTP 接口清单”文档。即使没有任何参考模板，也必须仅凭本提示词输出结构完整、接口准确、便于后续研发查阅和补齐能力边界的接口清单。

## 任务目标

请从项目代码中抽取所有已实现 HTTP 接口，并结合前端 API 调用识别未实现接口，生成一份面向研发协作的接口清单文档。

文档目标不是生成 Swagger，而是解释：

- 当前项目暴露了哪些 HTTP 接口。
- 每个接口属于哪个 Controller 和业务域。
- 每个接口的请求方式、路径、用途、主要入参、返回类型和登录要求。
- 哪些 Controller 存在但未暴露接口。
- 哪些接口被前端引用但当前后端未实现。
- 后续开发接口时应该优先关注哪些实现边界。

## 输入要求

生成前必须主动阅读并分析以下内容：

1. `Controller` 类：识别 `@RequestMapping`、`@GetMapping`、`@PostMapping`、`@PutMapping`、`@DeleteMapping` 等注解。
2. Controller 方法签名：识别 `@RequestBody`、`@RequestParam`、`@PathVariable`、`MultipartFile`、`HttpServletResponse` 等入参。
3. 返回类型：识别 `R<T>`、`Page`、`List`、文件流等返回形式。
4. `Filter`、`Interceptor`、`Security` 配置：识别登录拦截、白名单和权限要求。
5. `Entity`、`DTO`、`VO`：补充接口主要字段含义。
6. `ServiceImpl`：补充接口背后的关键业务规则，例如删除前校验、缓存清理、事务写入。
7. 前端 API 文件和页面代码：识别实际调用的接口，并与 Controller 对比，找出后端未实现的接口。
8. 配置文件：识别端口、上下文路径、接口文档入口、文件路径、缓存等信息。

如果项目中存在注释乱码或文档乱码，不要照抄乱码；应根据类名、字段名、接口路径和前端调用恢复接口含义。无法确认的内容必须标注“需进一步确认”，不要编造。

## 分析步骤

1. **统计 Controller**
   - 统计所有 Controller 类。
   - 区分“有效 Controller”和“空 Controller”。
   - 有效 Controller 指至少暴露一个 HTTP 方法映射。

2. **抽取已实现接口**
   - 拼接类级 `@RequestMapping` 和方法级映射，得到完整路径。
   - 识别 HTTP Method。
   - 识别方法用途，优先来自方法名、前端调用名、业务实现和字段含义。
   - 识别主要入参，按 Body、Query、Path、FormData 分类。
   - 识别返回类型。

3. **识别登录要求**
   - 从登录过滤器或安全配置中找出白名单。
   - 白名单内接口标记为“不需要登录”。
   - 其他接口标记为“需要登录”。
   - 如果权限粒度更细，例如角色、租户、组织、菜单权限，也要说明。

4. **按业务域分组**
   - 不要只按字母顺序罗列。
   - 按业务模块输出，例如员工管理、分类管理、商品管理、用户登录、地址、购物车、订单、文件等。
   - 每组先写 Controller 名称和路径前缀，再写接口表。

5. **补充关键业务说明**
   - 对有业务规则的接口，在说明中写清楚关键规则。
   - 示例：删除前校验关联数据、提交订单后清空购物车、上传文件使用 UUID、查询列表使用缓存。

6. **识别前端引用但后端未实现接口**
   - 扫描前端 API 文件中的 `url` 和 `method`。
   - 与 Controller 已实现接口对比。
   - 输出缺失接口表，包含来源文件、预期业务含义、当前状态。

7. **输出接口速查**
   - 汇总每个业务域、Controller、接口数和说明，方便快速定位。

## 表格字段说明

接口表必须包含：

- **接口路径**：使用 `` `METHOD /path` `` 格式。
- **说明**：说明接口业务用途和关键业务规则。
- **入参**：按 `Body`、`Query`、`Path`、`FormData` 分类描述主要参数。
- **返回**：写返回类型，例如 `R<User>`、`R<Page>`、文件流。
- **登录**：写“是”或“否”；如果存在角色或权限要求，写具体要求。

缺失接口表必须包含：

- **前端引用接口**：使用 `` `METHOD /path` `` 格式。
- **来源**：写具体前端文件或页面路径。
- **预期业务含义**：根据调用函数名、页面文案和接口路径推断。
- **当前状态**：通常为“未实现”；如果后端路径存在但方法不匹配，写“方法不匹配”。

## 写作风格要求

1. 使用中文，表达直接、准确、工程化。
2. 不要照抄乱码注释。
3. 不要只输出接口路径，要补充业务含义、入参、返回和登录要求。
4. 类名、路径、字段名、DTO 名称、返回类型使用反引号。
5. 对复杂业务规则写在接口说明中，不要另起大段代码说明。
6. 对无法确认的参数字段，写“主要字段”或“需进一步确认”。
7. 不要把前端静态资源路径当作业务接口，但登录白名单中可以说明。
8. 不要把 Swagger、Knife4j、静态资源接口计入核心业务接口数，除非用户明确要求。
9. 接口数量必须与表格中已实现接口数一致。
10. 前端引用但后端未实现的接口必须单独列出，不能混入已实现接口。

## 质量检查清单

输出前逐项检查：

- 是否扫描了所有 Controller。
- 是否完整拼接了类级路径和方法级路径。
- 是否区分了 GET、POST、PUT、DELETE。
- 是否统计了空 Controller。
- 是否标注了登录白名单和需要登录的接口。
- 是否说明了统一返回结构。
- 是否将接口按业务域分组。
- 是否列出了前端引用但后端未实现的接口。
- 已实现接口数量是否与接口表逐项相加一致。
- 缺失接口数量是否与缺失接口表逐项相加一致。
